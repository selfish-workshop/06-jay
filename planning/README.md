# 기획 (Planning) — 퇴근을 향하여

이 폴더는 "무엇을 만들 것인가"의 상세를 담습니다.
기획 파트리드로서 PM 초안을 바탕으로 **디자인팀·개발팀에게 바로 넘길 수 있는 수준**까지 구체화한 문서 모음입니다.

## 문서 구성

| 파일 | 내용 |
|---|---|
| [`01-feature-spec.md`](./01-feature-spec.md) | 기능 명세 (F1~F6 화면별 동작 정의) |
| [`02-user-scenarios.md`](./02-user-scenarios.md) | 사용자 시나리오 (용사/보스/발표자 3명) |
| [`03-mvp-scope.md`](./03-mvp-scope.md) | MVP 범위 (Must/Should/Won't have) |
| [`04-priorities.md`](./04-priorities.md) | 작업 우선순위 (P0~P3) 및 팀별 넘기기 가이드 |

## 읽는 순서 (추천)

1. **MVP 범위** (`03-mvp-scope.md`) — 데모에서 뭐가 필수인지 먼저 파악
2. **기능 명세** (`01-feature-spec.md`) — 각 기능이 어떻게 동작하는지
3. **사용자 시나리오** (`02-user-scenarios.md`) — 실제로 어떻게 쓰이는지
4. **우선순위** (`04-priorities.md`) — 어떤 순서로 만들지

## 다음 파트로 넘길 때

- **디자인 팀**: `01-feature-spec.md`의 화면 구성 + `04-priorities.md`의 P0 화면 5개부터 와이어프레임 시작
- **CX 팀**: `02-user-scenarios.md`의 시나리오 1, 2를 페르소나·여정맵으로 확장
- **개발 팀**: `03-mvp-scope.md`의 DoD와 기술 스택 권장 참고
- **마케팅/전략 팀**: PRD.md의 해당 섹션과 연결

## 핵심 원칙

1. **직원 부담 최소화** — 업무 입력이 게임처럼 가벼워야 함
2. **리더 가시성 극대화** — 팀원에게 묻지 않고도 파악 가능
3. **재미 ≠ 감시 툴** — 유머는 있되 관리자용 감시 느낌 금지
4. **발표 5분 안에 완결** — 데모 시나리오가 끊김 없이 흘러가야 함
