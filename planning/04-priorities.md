# 작업 우선순위

디자인팀·개발팀에게 넘길 때 참고할 우선순위 기준. 숫자가 낮을수록 먼저.

---

## P0 — 크리티컬 패스 (이게 없으면 데모 불가)

이 순서대로 만들면 **가장 짧은 데모**가 가능한 최소 경로.

| # | 항목 | 이유 |
|---|---|---|
| 1 | 모드 선택 화면 | 서비스 진입점, 컨셉 첫인상 |
| 2 | 캐릭터 생성 = 업무 입력 | 컨셉의 가장 중요한 UX 트릭 |
| 3 | 퀘스트 메인 화면 (용사모드) | 입력이 어떻게 시각화되는지 보여주는 핵심 화면 |
| 4 | 퀘스트 진행률 업데이트 모달 | "업무가 게임처럼 관리된다" 실체 |
| 5 | 보스 대시보드 | 양면성(Dual mode) 컨셉 증명 |

**순서 고정 이유**: 1→5로 가면 데모가 자연스럽게 흐른다. 중간에 하나라도 빠지면 다음이 안 돌아감.

---

## P1 — 컨셉 완성도 (있어야 발표 임팩트)

| # | 항목 | 이유 |
|---|---|---|
| 6 | 오프닝 시퀀스 | 세계관 몰입 + "재밌는 서비스" 첫인상 |
| 7 | Skip 버튼 | 없으면 데모 반복 시연 시 피곤. 편의 필수 |
| 8 | 막힘 상태 + 메모 | 보스모드의 "이 팀원 도움 필요" 시나리오 핵심 |
| 9 | 더미 팀원 데이터 | 보스 대시보드가 "혼자 쓰는 화면"으로 보이면 안 됨 |
| 10 | 데모 초기화 버튼 | 발표 후 다시 시연하려면 필수 |

## P3 — 의도적으로 뺀 것

데모 범위 밖. 언급만 해두고 구현 안 함.

- 실시간 동기화
- 로그인
- 서버 연동
- 알림
- 모바일 최적화 (데스크탑 데모만)
- 경험치/레벨 시스템

---

## 팀별 넘기기 우선순위

**기획 → 디자인 팀**: P0의 5개 화면 구조 + P1의 오프닝 시퀀스 연출 톤
**기획 → CX 팀**: 시나리오 1, 2 (용사/보스) → 페르소나·여정맵으로 확장
**기획 → 개발 팀**: P0 → P1 → P2 순서로 시간 배분, P3는 명시적으로 제외
**기획 → 마케팅 팀**: "퇴근이 1시간 빨라진다" 핵심 메시지 + 데모 영상 소스
**기획 → 전략 팀**: P3의 실적관리 확장을 장기 방향으로만 채택

---

## 리스크

- **오프닝 시퀀스가 P1인 이유**: 컨셉 임팩트는 크지만 없어도 데모 성립. 개발 시간 부족 시 제일 먼저 버릴 수 있는 항목.
- **보스 대시보드 더미 데이터**: 이게 어색하면 "혼자 쓰는 화면"처럼 보임. 진행률·막힘 상태를 다양하게 섞어야 함.
- **localStorage 의존**: 브라우저 바꾸면 데이터 날아감. 발표 현장에서 같은 브라우저·같은 창 유지 필수.
